# Personal Memory System V1 Delivery Record

## 为什么写这份文档

`personal-memory-system-v1-scope` 定义了 V1 应该做什么。

但随着实现推进，V1 已经不只是一个最小 scaffold。

它已经形成了一个可运行、可跨设备访问、可持续存储、可做 retrieval-aware reply、并且已经完成云上部署迁移的完整闭环。

所以这里需要一份单独文档，回答四个问题：

- V1 实际做成了什么
- V1 到底验证了什么
- 当前产品重点已经收敛到哪里
- 真实使用中暴露了哪些问题，哪些问题应延续到 V1.5 / 后续版本继续处理

这份文档默认会持续更新。

它不是 roadmap，也不是宣传页。

它是交付记录。

## V1 的原始目标

V1 一开始只验证三件事：

- conversation persistence
- retrieval-aware replies
- daily reflection

这一点没有变。

真正重要的是：

V1 不是为了做一个能力很多的助手壳，而是为了验证“对话是否真的可以成为个人连续性的主入口”。

## V1 实际已经完成的内容

## 1. 后端闭环已经完整跑通

当前 V1 已经不是纸面上的 memory backend，而是一个能实际运行的 Node.js + SQLite 服务。

已经完成的后端能力包括：

- 对话、消息、profile facts、memory events、reflections 的 SQLite 持久化
- conversation create / rename / delete / restore
- OpenAI-compatible chat completions 接口
- SSE 流式回复路径
- request id、auth、rate limit、health、metrics 等生产防护
- backup / restore 脚本
- 外部 conversation key 和 message key 的映射与去重
- cold start 后的 client state restore

这意味着 V1 已经具备一个真正可被原生客户端消费的后端底座，而不只是本地实验脚本。

## 2. 记忆最小闭环已经实现

V1 最核心的事情，是每轮对话不再只是“回复完就结束”。

当前系统已经具备以下记忆路径：

1. 用户消息和助手消息写入原始消息层
2. 每轮对话结束后抽取 memory events 和 profile facts
3. 新消息到来时构建 retrieval context
4. retrieval context 注入到模型调用中
5. daily reflection 按日期生成总结与 open loops

这套链路虽然还不够精细，但闭环已经成立。

也就是说，V1 已经从“普通聊天记录器”跨到了“会把对话重新喂回后续回复”的系统。

## 3. Jarvis 已经从 generic assistant 收敛到更明确的人格方向

在实现过程中，V1 还完成了一次很关键的 prompt 架构收敛。

当前系统不再把 Jarvis 定义成“memory assistant”本身，而是拆成两层：

- App 侧 system prompt 负责 Jarvis 的人格、语气、判断方式
- Backend 侧 system prompt 负责 memory 的使用约束

这次收敛解决了一个很重要的问题：

记忆应该是 Jarvis 的后台能力，而不是 Jarvis 的身份。

经过几轮 prompt 修改之后，Jarvis 的语气、回复结构、对话内部节奏，已经明显更接近目标形态。

目前可以认为：

- 语气基线已基本校准
- 结构基线已基本校准
- prompt 大方向已经从“客服型记忆助手”切回“高带宽对话对象”

## 4. Apple 双端原生客户端已经形成可用主入口

V1 当前的主要前端入口不是 Web 壳，而是原生 Apple 客户端。

已经完成的客户端能力包括：

- iPhone 和 macOS 双端 SwiftUI 客户端
- conversation list 与 chat detail
- streaming message rendering
- image attachment picking
- backend configuration 与健康检查
- 本地缓存与冷启动恢复
- onboarding
- 后台健康状态展示
- 自动轮询同步和状态恢复
- App 内 system prompt 编辑与保存

这件事的意义不是“有了 UI”。

真正的意义是：

V1 已经进入真实使用状态，而不是只停留在接口联调阶段。

## 5. 跨设备使用链路已经打通

当前 V1 已经支持：

- iPhone 和 macOS 双端读取同一份对话与记忆数据
- 通过 state restore 和轮询同步保持一致性
- 同一后端被多个原生端长期消费

这个能力先在本地局域网形态下被验证，随后又被推进到云上部署形态。

因此现在已经不是“只在一台 Mac 上做联调”，而是：

- 局域网多端访问链路已经验证过
- 公网 HTTPS 访问链路也已经验证过
- Apple 双端都已经能够连接同一份云上 backend

这意味着“同一个 Jarvis 被多个原生端消费”的验证已经成立，而且不再依赖手机和 Mac 处于同一 Wi-Fi。

## 6. 云上部署与 always-online 路径已经完成验证

V1 文档写完之后，又完成了一次非常关键的运行形态迁移：

- 从本地 Mac 自托管
- 迁移到 云主机
- 使用 Docker Compose + Caddy + HTTPS 提供公网访问

这件事的意义非常大。

因为它验证的已经不只是“memory backend 能不能跑”，而是：

- Jarvis 能不能脱离本地局域网环境稳定工作
- Apple 客户端能不能通过公网长期连接同一个后端
- 系统能不能具备真正的 always-online 运行形态

迁移过程中，实际踩到并解决了几类关键问题：

1. iPhone 对公网 HTTP 不可用，必须切到 HTTPS。
2. 仅有云主机并不够，必须有稳定 hostname 与 TLS。
3. 上游模型提供方的区域可用性会影响后端是否真正可用。

最终走通的部署路径是：

- 使用 云服务器实例（Ubuntu）
- 使用 Docker 部署 Node.js + SQLite backend
- 使用 Caddy 反向代理并申请 HTTPS 证书
- 在没有正式域名阶段，使用 `sslip.io` 作为临时 hostname
- 最终将可用生产节点落在可用区域

这里最重要的真实结论有三个：

- V1 的 backend 已经被验证可以作为公网长期服务存在
- Apple 双端已经不必依赖本地 Mac 常开才能访问同一个 Jarvis
- 区域与 provider 可达性已经被证明是生产部署的真实约束，而不是理论问题

## 7. 生产部署链路已经补齐到可复用级别

在这次云迁移之后，V1 的运维能力不再只是“本地能启动”。

已经具备的实际部署资产包括：

- `.env` 驱动配置
- Dockerfile
- `compose.yml` 与 `compose.prod.yml`
- Caddy HTTPS 反向代理配置
- 云部署 runbook
- 基础服务器 bootstrap 脚本

这意味着当前仓库已经不只是一个开发仓库。

它已经具备：

- 本地开发路径
- 本地自托管路径
- 云上生产部署路径

而且这些路径已经经过真实验证，而不是仅停留在文档层。

## 8. 仓库与文件迁移已经完成

在部署迁移之外，这一阶段还完成了一次非常重要的仓库与文件迁移：

- 重做版工作区已经从旧的 `openclaw` 路径中分离出来
- 当前主工作区已经独立为 `Jarvis`
- 旧的 OpenClaw 本地进程、运行目录与启动项已经被清理
- OpenClaw 版本又被单独整理成一个脱敏后的公开参考仓库，用于保留产品、工程和架构资产

这件事的重要性在于：

- 当前重做版 Jarvis 不再继续背着 OpenClaw 的目录和运行语义前进
- 历史资产得到了保留，但不会再和当前主线仓库混在一起
- 产品主线、历史归档、公开参考实现三者的边界已经清晰

## 9. 运维边界已经初步补齐

V1 不只是能跑，也已经补上了最低限度的运维能力：

- `.env` 驱动配置
- Docker / Compose 部署路径
- Bearer token auth
- 简单 rate limiting
- health / metrics
- SQLite backup / restore

这意味着当前版本已经可以作为一个长期本地服务持续运行，而不需要每次都靠手工临时启动和临时调试。

在云迁移之后，这个判断还应再往前走一步：

当前版本已经可以作为一个长期云上服务持续运行，而不必再默认依赖本地 Mac 常驻。

## V1 实际验证了什么

到目前为止，V1 已经实际验证了六件事。

## 1. 对话可以稳定成为主数据源

对话本身已经足够支撑最小连续性系统。

至少在当前阶段，不需要先引入图片、定位、被动 life-log，系统也能开始积累连续性。

## 2. retrieval-aware reply 是成立的

系统已经不是单纯重放历史。

它会在每轮回复前取回少量相关上下文，把 profile facts、memory events、recent messages、reflection 重新组合进回答过程。

这说明 V1 的核心假设是成立的：

少量而准确的记忆检索，比大段原始历史重放更重要。

> **⚠️ 重要修正（2026-03-30）：** 后续代码审查发现 V1 的检索实现存在严重的中文失聪缺陷。`retrieval.js`、`database.js` 和 `reflection-service.js` 中的三个 `tokenize()` 函数均使用 `/[^a-z0-9]+/` 正则——只保留英文字母和数字，中文字符被完全过滤。此外 `extraction.js` 的 11 条 profile fact 匹配规则全部是英文正则。由于用户主要使用中文对话，这意味着：
>
> - token-overlap 检索对中文消息的得分永远为 0
> - 中文对话内容在检索中不可见
> - 中文表达的个人事实无法被提取为 profile facts
>
> 因此 **"retrieval-aware reply 是成立的"这个结论需要打折扣**——架构闭环成立，但检索质量在中文场景下基本无效。这是 V1 到 V1.5 之间最紧迫的修复项。详见 `/design/retrieval-rewrite-plan`。

## 3. 原生客户端是正确入口

当前真实使用已经证明：

- 原生客户端可以带来更低摩擦的进入方式
- 设置、缓存、同步、健康检查这些基础能力，放在原生端是合理的
- “先有可用入口，再继续打磨连续性”这条路是对的

## 4. Jarvis 的产品重点不是“助手能力”，而是“对话质量”

实现和调试过程里，最影响体验的不是工具多不多，也不是 memory database 看起来多不多。

最影响体验的是：

- Jarvis 是否真的在和用户对话
- 是否有判断力
- 是否不机械
- 是否把 memory 当背景能力而不是自我介绍

这意味着后续版本的主要工作，仍然应该继续围绕 conversation quality 和 continuity quality，而不是 feature surface 扩张。

## 5. 记忆功能还没有被充分验证

这一点同样重要。

虽然记忆链路已经实现，但当前真实对话量还不够大，不能说明记忆系统已经成熟。

> **补充（2026-03-30）：** 记忆功能未被充分验证还有一个比对话量更根本的原因——检索模块对中文失聪（见上文第 2 点修正），导致用中文积累的记忆根本无法被检索到。在修复 retrieval 之前，"记忆是否真正提升连续性"这个命题无法被有效验证。

现在只能说：

- 记忆架构闭环成立了
- 数据已经在积累
- 基本读写都通了
- 但真实长期质量还没有经过充分压力测试

所以 V1 不能被误判成“记忆已经完成”。

更准确的说法是：

V1 已经把记忆系统带入可测试阶段。

## 6. 云部署已经从“可想象”变成“已跑通”

这次从本地迁到云服务器的过程，实际上又验证了一层以前还没有真正落地的能力：

- Jarvis backend 可以脱离本地设备长期在线
- Apple 客户端可以在公网下访问同一后端
- HTTPS、认证、反向代理和生产运行路径已经形成完整闭环

更重要的是，迁移过程还验证了一个生产事实：

基础设施地区选择会直接影响上游模型服务是否可用。

这不是“部署细节”，而是产品可运营性的组成部分。

## 当前阶段的产品重点

V1 当前最重要的重点，不再是“继续加能力”，而是三件事：

- 提高前端稳定性
- 保持 Jarvis 的对话质感不回退
- 在更多真实使用中继续验证 memory quality

除此之外，现在还多了一条新的长期重点：

- 把云上单后端形态稳定下来，避免本地后端与云后端双轨并存造成运行和数据源混乱

换句话说，当前最值钱的不是扩张，而是稳住。

具体来说：

- 后端基础闭环已经足够用来继续收集真实使用数据
- prompt 大方向已经基本校准，不应再轻易退回“memory assistant”口吻
- 下一阶段的真实瓶颈开始更多落在产品稳定性和使用摩擦上

## 当前已知使用体验反馈

以下内容来自当前真实使用反馈，应被视为后续版本的重要输入。

## 一、前端

双端经过优化之后，整体体验已经可用，UI 的舒适感也比较高。

但稳定性仍然不够，仍然有一批明确的小 bug 和交互缺口：

1. Mac 端黑夜模式下，对话框文字会变成白色，和文本框背景接近，导致可读性出问题。
2. Jarvis 的对话气泡名称现在还是“助手”，需要改成“Jarvis”。用户自己的气泡名称现在是“你”，这也是错的，应改成“我”。
3. Mac 端在设置页打开时，无法通过右键点击 icon 关闭 app。
4. 双端都没有“一键跳到对话末尾”按钮，而且每次打开对话默认停在顶部。随着对话变长，这会越来越不友好。
5. 没有“重新生成回复”功能。
6. iOS 端当前是回车换行、点击发送，这个行为正确。Mac 端需要改成回车发送，`Command + Enter` 换行。
7. 当前存在一个很大的流式渲染问题：当用户消息较长时，自己的气泡会短暂消失；Jarvis 回复较短时会先显示部分内容，然后气泡消失，等完整回复结束后再出现；Jarvis 回复较长时，气泡会直接消失几秒，直到回复结束才重新出现。
8. 双端 app icon 的白底面积偏大，logo 本体偏小，视觉上需要稍微放大一些。
9. 输入框存在稳定性问题。有时输入过程中对话会突然漂移到前面，且似乎还有其他动作也会触发输入框不稳定，但触发条件目前还没有被完全定位。

## 二、部署与运行形态

云迁移之后，运行形态已经从“本地 Mac 提供服务”切到“云上单后端 + 本地客户端”。

这带来了两个新的经验：

1. 正式使用时，不应再让本地后端和云后端同时作为主服务存在，否则容易混淆真实数据源。
2. 客户端仍然可能保留旧 bundle id 或旧配置痕迹，因此应用身份清理和后端地址校准仍然属于真实交付的一部分。

## 三、Jarvis 的语气与回复结构

经过几轮 prompt 修改之后，Jarvis 的语气和回复结构已经基本校准。

经过对话内部调试，它已经越来越接近目标中的 Jarvis 形态。

这部分目前不是主要问题源，后续应该以稳住为主，而不是大幅重写。

## 四、记忆功能仍需更多真实使用

目前对话量仍然偏少，还没有真正测到 memory system 的上限和问题密度。

所以记忆功能现在仍处于“需要继续积累使用体验”的阶段。

后续反馈重点应包括：

- Jarvis 是否会在正确的时候自然调起旧信息
- profile facts 是否会变脏、重复或失真
- memory events 是否过于机械
- daily reflection 是否真的能提升后续连续性

## 这些反馈应该如何进入后续版本

这批反馈不要求全部在 V1.5 内解决。

更合适的处理方式是：

- V1.5 优先修那些直接阻碍持续使用的稳定性问题
- V2 再系统性清理当前前端交互债务和完整体验问题

也就是说：

V1.5 的目标仍应偏向“让连续性更可信”。

V2 再把当前已经暴露出的交互和产品完成度问题集中收尾。

## 当前对 V1 的总体判断

V1 可以认为已经完成了第一阶段最重要的事情：

- 后端闭环成立
- 原生端可用
- 云上单后端形态已经跑通
- Jarvis 的基本人格方向收敛成功
- 真实使用已经开始
- 工作区和历史代码边界已经理清

但 V1 也很清楚地暴露出：

- 前端稳定性仍然不够
- 记忆质量还远没有经过充分验证
- 产品真正难的部分已经从“能不能跑”转向“能不能长期好用”
- 云上运行、客户端配置、应用身份和数据源一致性现在也进入长期维护范围

所以对 V1 最准确的定义不是“完成版”。

而是：

V1 已经把系统推进到了真实可用、可测试、可继续打磨的阶段。

这正是它应该完成的任务。
